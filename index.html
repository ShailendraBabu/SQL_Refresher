DDL: Defining Your Database Schema

DDL commands are essential for building and maintaining the structural integrity of your database. They define how data is organized and stored.

CREATE: Building Database Objects

The `CREATE` statement establishes new database objects like tables. You define column names, data types, and constraints.

CREATE TABLE employees ( id INT AUTO_INCREMENT PRIMARY KEY, first_name VARCHAR(50), last_name VARCHAR(50), hire_date DATE );

SQL Data Types:

  • Numeric: INT, DECIMAL, FLOAT
  • Character/String: VARCHAR, CHAR
  • Date/Time: DATE, TIME, DATETIME

SQL Constraints:

  • NOT NULL: Ensures a column cannot store NULL.
  • UNIQUE: All values in a column are distinct.
  • PRIMARY KEY: Unique identifier for each row (NOT NULL + UNIQUE).
  • FOREIGN KEY: Links two tables, enforcing referential integrity.
  • CHECK: Ensures values satisfy a condition.
  • DEFAULT: Provides a default value if none specified.
  • INDEX: Speeds up data retrieval.

ALTER: Modifying Existing Structures

The `ALTER TABLE` statement changes an existing table's structure, like adding or dropping columns and constraints.

Add Column:

ALTER TABLE employees ADD department_id INT;

Drop Column:

ALTER TABLE employees DROP COLUMN hire_date;

Modify Column Type:

ALTER TABLE employees MODIFY COLUMN first_name VARCHAR(100);

Add Foreign Key Constraint:

ALTER TABLE Orders ADD FOREIGN KEY (PersonID) REFERENCES Persons(PersonID);

DROP, TRUNCATE, RENAME: Managing Table Lifecycle

These commands manage the existence and naming of tables.

DROP TABLE:

Completely removes a table (structure + data).

DROP TABLE old_employees;

TRUNCATE TABLE:

Removes all rows from a table, but keeps the structure. Faster than DELETE for all rows.

TRUNCATE TABLE temporary_logs;

RENAME TABLE:

ALTER TABLE old_table_name RENAME TO new_table_name;

DML: Interacting with Your Data

DML commands are used to add, modify, and remove the actual data within your database tables.

INSERT: Adding Data

The `INSERT` command adds new rows of data into a table.

Single-row INSERT:

INSERT INTO employees (first_name, last_name) VALUES ('Alice', 'Smith');

Multi-row INSERT (from another table):

INSERT INTO new_employees (id, first_name) SELECT employee_id, name FROM old_employee_records WHERE status = 'Active';

UPDATE: Modifying Data

The `UPDATE` statement changes existing data in one or more rows. The `WHERE` clause is crucial to specify which rows to modify.

UPDATE Products SET Price = Price * 1.10, Stock = Stock - 5 WHERE Category = 'Electronics';

DELETE: Removing Data

The `DELETE` statement removes rows from a table. Use the `WHERE` clause to specify conditions.

Delete specific rows:

DELETE FROM Orders WHERE OrderDate < '2023-01-01';

Delete all rows (without WHERE):

DELETE FROM TempTable;

Basic `SELECT`

Retrieve all (`*`) or specific columns from a table.

Select all columns:

SELECT * FROM Customers;

Select specific columns:

SELECT CustomerName, City FROM Customers;

Filtering with `WHERE`

Extract only records that fulfill a specified condition using comparison and logical operators.

Example with comparison operator:

SELECT * FROM Products WHERE Price > 50;

Example with logical operator (`AND`):

SELECT * FROM Orders WHERE OrderDate >= '2024-01-01' AND TotalAmount > 100;

Sorting with `ORDER BY`

Sort the result-set in ascending (`ASC`, default) or descending (`DESC`) order by one or more columns.

Sort by one column descending:

SELECT ProductName FROM Products ORDER BY ProductName DESC;

Sort by multiple columns:

SELECT City, CustomerName FROM Customers ORDER BY City ASC, CustomerName ASC;

Limiting Results (`LIMIT`/`TOP`)

Restrict the number of rows returned by a query, useful for pagination or top N results.

`LIMIT` (MySQL/PostgreSQL):

SELECT ProductName FROM Products ORDER BY Price DESC LIMIT 10;

`TOP` (SQL Server):

SELECT TOP 5 CustomerName FROM Customers ORDER BY CustomerID DESC;

Aggregate Functions and Grouping

Aggregate functions perform calculations on a set of rows, returning a single summary value. `GROUP BY` groups rows with identical values for aggregation, and `HAVING` filters these groups.

Common Aggregate Functions:

  • COUNT(): Number of rows.
  • SUM(): Total sum of numeric values.
  • AVG(): Average of numeric values.
  • MIN(): Lowest value.
  • MAX(): Highest value.

Example with `GROUP BY` and `HAVING`:

SELECT Country, COUNT(CustomerID) AS CustomerCount FROM Customers GROUP BY Country HAVING COUNT(CustomerID) > 5;

JOINs: Combining Tables

JOINS are fundamental to relational databases, allowing you to retrieve data from multiple tables in a single query based on related columns.

INNER JOIN

Returns records that have matching values in both tables.

LEFT JOIN

Returns all records from the left table, and the matched records from the right table.

RIGHT JOIN

Returns all records from the right table, and the matched records from the left table.

FULL OUTER JOIN

Returns all records when there is a match in either left or right table.

Example of an `INNER JOIN` to get order information along with customer names.

SELECT Orders.OrderID, Customers.CustomerName FROM Orders INNER JOIN Customers ON Orders.CustomerID = Customers.CustomerID;

Advanced SQL Constructs

Beyond basic operations, these constructs enable more sophisticated data retrieval and manipulation, crucial for complex problem-solving.

Subqueries (Nested Queries)

A query embedded within another SQL query. They are powerful for complex filtering or when criteria depend on another query's result.

Example: Products more expensive than average:

SELECT ProductName, Price FROM Products WHERE Price > (SELECT AVG(Price) FROM Products);

CTEs (Common Table Expressions)

A temporary, named result set defined using the `WITH` clause. Improves readability, modularity, and allows for recursive queries.

Example: Products in 'Beverages' category using CTE:

WITH BeverageCategories AS ( SELECT CategoryID FROM Categories WHERE CategoryName = 'Beverages' ) SELECT ProductName FROM Products WHERE CategoryID IN (SELECT * FROM BeverageCategories);

Performance & Best Practices

Optimizing SQL queries is crucial for building scalable and efficient applications. Understanding how databases execute queries and applying best practices can dramatically improve performance.

The Power of Indexing

Indexes are crucial for performance. They allow the database to find data quickly without scanning the entire table, much like a book's index. Notice the dramatic improvement in query time.

Types of Indexes: Clustered (physical order), Non-Clustered (separate structure), Composite (multiple columns), Covering (all query columns). Trade-offs: Faster reads, but slower writes (INSERT/UPDATE/DELETE) due to index maintenance.

Query Tuning Quick Wins

Be Specific with `SELECT`

Avoid `SELECT *`. Only request the columns you actually need. This reduces data transfer and enables index-only scans.

‚úì

Use Sargable Predicates

Conditions in `WHERE` clauses that allow the database to use an index efficiently. Avoid functions on indexed columns.

‚úó

Avoid `WHERE YEAR(order_date) = 2024`

Instead, use: `WHERE order_date >= '2024-01-01' AND order_date < '2025-01-01'`

‚úì

`JOIN`s over Complex Subqueries

When possible, use `JOIN`s. They are often more efficient as the database optimizer is better at handling them.

‚úì

Batch Operations

For large `DELETE`/`UPDATE` operations, perform them in smaller batches to prevent long-running transactions and locking.

‚úì

Schema Design (Normalization vs. Denormalization)

Normalize for data integrity and efficient writes; denormalize for faster reads in specific analytical scenarios.
